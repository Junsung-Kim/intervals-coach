# 이 템플릿에 맞는 사람 / 기대와 한계

## 잘 맞는 경우

- **Intervals.icu**에 본인 운동·웰니스·이벤트를 일상적으로 쌓고 있고, API로 끌어와 **LLM 코칭**에 쓰고 싶은 경우.
- **철인 3종(또는 멀티스포츠)** 준비로 수영·러닝·사이클을 함께 다루고, **CTL/ATL/TSB** 같은 피로도 지표와 스케줄을 한곳에서 보고 싶은 경우.
- GitHub·Cloudflare를 **한 번 설정**해 두고, 이후에는 Actions로 자동 동기화되는 흐름을 원하는 경우.

## 단일 종목만 할 때 (러닝만, 자전거만 등)

템플릿은 **그대로 동작**한다. 다만 LLM이 내리는 해석의 질은 **Intervals에 들어온 데이터의 풍부함**에 좌우된다.

| 구분 | 있으면 좋은 것 | 없을 때의 한계 |
|------|----------------|----------------|
| **러닝** | **threshold pace(T-pace)** 등 역치 페이스가 프로필/요약에 반영됨 | 페이스·심박만 있는 기록은 존/강도 해석이 넓어짐 |
| **사이클** | **FTP**, 파워 기반 활동, 파워 존 타임 | 심박만·거리만이면 NP·부하 모델 해석이 제한적 |
| **수영** | CSS 등 수영 역치·풀 세션 기록 | 기록이 적으면 “부하 균형·회복” 위주만 안전하게 다루는 편이 좋음 |

## 장비·데이터 전제

- **파워미터**(사이클): FTP·파워 존 분석에 유리. 없어도 사용은 가능하나 **파워 기반 처방**은 신뢰도가 떨어진다.
- **러닝**: GPS·심박·페이스가 꾸준히 쌓일수록 역치·존 해석이 안정적이다.
- **수영**: 워치/수동 입력 등 Intervals에 들어가는 방식은 사용자마다 다름. 강습·기술은 현장 코치 영역으로 두고, **이 템플릿은 주로 부하·일정·회복 데이터**를 다룬다고 보면 된다.

## Intervals에서 채워 두면 좋은 설정

- 프로필·종목별 섹션에 **Cycling: FTP**, **Running: threshold_pace**, **Swimming: CSS** 등 역치 값을 최신으로 유지한다.
- 목표·이벤트·스케줄 제약은 `summary` 텍스트의 `[goals]`, `[schedule]` 등에 반영되도록 Intervals에서 관리한다.

자세한 용어·이론은 [knowledge/](knowledge/README.md)를 참고한다.
